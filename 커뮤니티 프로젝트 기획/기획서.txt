## 회원가입 페이지 [구현 완료]

레이아웃
- 로그인 페이지와 동일한 2단 분할 구조
  왼쪽(42%): 보라 그라데이션 일러스트 패널 + 토끼 마스코트
  오른쪽(58%): 스크롤 가능한 회원가입 폼 (최대 높이 90vh)
- 모바일: 단일 컬럼, 상단 토끼 소형 표시

구성 요소
- 이메일 입력란
- 닉네임 입력란
- 비밀번호 입력란
- 비밀번호 확인 입력란
- 회원가입 버튼
- 관심 카테고리 선택 칩 (선택 사항)

회원가입 입력 UX
- 비밀번호 강도 표시
  약함 / 보통 / 안전함 / 매우 안전
  (LinearProgress 바로 시각화)

- 비밀번호 일치 실시간 표시
  ✔ 비밀번호가 일치합니다
  ❌ 비밀번호가 일치하지 않습니다

- 비밀번호 보기 버튼 제공

- 입력창 상태 디자인
  기본 상태: 연보라 border (#D8CFF0)
  focus 상태: primary.main border (#9B82CC)
  error 상태: error.main border

- 관심 카테고리 칩 선택 시
  단색 primary.main 배경 (그라데이션 없음)

디자인 요소
- 로그인 페이지와 동일한 감성 유지
- 연보라 계열 단색 버튼 사용 (그라데이션 없음)
- 입력폼은 깔끔하고 여백 있게 배치
- 배경 이미지 위에 카드가 떠 있는 느낌

## 방명록 페이지 [구현 완료]

페이지 소개
- Winter Log에 방문자가 응원 메시지를 남기는 공간
- URL: /guestbook

작성 폼 (로그인 필수)
- 오늘의 기분 이모지 선택
  🌱 ☁️ 🌙 ✨ 💜 ☕ 🐰 중 선택
- 응원 온도 (별점 1~5개)
  - 1개: 따뜻한 마음이 전해져요 ☁️
  - 2개: 포근한 응원 감사합니다 🌱
  - 3개: 별빛 같은 응원이 도착했어요 ✨
  - 4개: 마음이 따뜻해지는 응원이에요 💜
  - 5개: 온 마음을 다한 응원 감사합니다 🌙
- 메시지 입력 (최대 300자)
- 작성 후 1페이지로 이동 (최신 방명록 확인)

방명록 목록
- 이모지 + 작성자 아바타/닉네임 + 별점 + 메시지 + 작성 시간
- 5개씩 서버사이드 페이지네이션
- "총 N개의 응원" 카운트 표시

- 작성 폼 대신 로그인 안내 메시지 표시

## 마이페이지 [구현 완료]

레이아웃 (2단 구조)
- 왼쪽 컬럼 (md:4): 프로필 카드 + 최근 작성 게시글
- 오른쪽 컬럼 (md:8): 공부 잔디 → 오늘의 목표 → 할 일 관리

프로필 영역
- 프로필 이미지 (카메라 아이콘 클릭 → 파일 선택 → Supabase Storage 업로드)
  - avatars/{user_id}/avatar.{확장자} 경로로 저장
  - 업로드 즉시 미리보기 반영 (blob URL + 캐시 버스팅)
- 닉네임
- 자기소개
- 활동 점수 표시
- 관심 카테고리 (체크박스형 칩)
- 프로필 수정 저장 버튼

공부 성장 영역
- 공부 잔디 (52주 × 7일 히트맵)
  - 게시글 작성 시 해당 날짜 자동 활성화
  - post_count 기반으로 색상 강도 표시
- 연속 공부일 표시 (study_streak)
- 오늘의 공부 목표 입력 및 저장

할 일 관리 영역

# 기술 스택

Frontend
- React 19
- Vite 7
- React Router v7 (HashRouter - GitHub Pages 호환)

Styling
- MUI (Material UI) v9
- MUI sx 속성 기반 스타일링
- 커스텀 테마 (getTheme 함수로 라이트/다크 분리)

마크다운
- react-markdown
- remark-gfm

Backend
- Supabase (Auth + Database + Storage, 별도 서버 없음)
- RLS(Row Level Security) 전체 테이블 적용
- Supabase Auth 트리거로 회원가입 시 자동 프로필 생성
- SECURITY DEFINER RPC 함수로 카운터 관리 (view_count, like_count, comment_count)
- Supabase Storage: avatars 버킷 (프로필 이미지 저장)

상태관리
- Zustand (persist 미들웨어 사용)
- themeMode만 localStorage 저장, 유저 세션은 저장 안 함

배포
- GitHub Pages (GitHub Actions 자동 배포)
- 브랜치: main 푸시 시 자동 빌드 및 배포
- base path: /my-community/
- Supabase Keep-Alive 워크플로우 (주 3회 ping, 무료 플랜 자동 일시정지 방지)

== Winter Log DB 구조 ==

테이블 이름 규칙
- 모든 테이블 이름 앞에는 사이트명인 winterlog_ 를 붙여서 생성한다.

--------------------------------------------------

1. winterlog_users 테이블
- 회원 정보 저장
- 회원가입, 로그인, 마이페이지에서 사용

필드:
- user_id: 사용자 번호
- email: 이메일
- password: 비밀번호
- nickname: 닉네임
- profile_image: 프로필 이미지 URL (Supabase Storage avatars 버킷)
- bio: 자기소개
- interest_categories: 관심 카테고리
- theme_mode: light / dark
- activity_score: 활동 점수
- study_streak: 연속 공부일
- created_at: 가입일

--------------------------------------------------

2. winterlog_posts 테이블
- 게시글 정보 저장
- 게시물 목록, 게시물 상세 페이지에서 사용

필드:
- post_id: 게시글 번호
- user_id: 작성자 번호
- title: 게시글 제목
- content: 게시글 내용 (Markdown 형식)
- category: 카테고리
- status_tag: 공부중 / 질문 / 해결완료 / 회고 / 팁공유
- image_url: 이미지 주소
- like_count: 좋아요 수
- upvote_count: 업보트 수
- downvote_count: 다운보트 수
- comment_count: 댓글 수
- view_count: 조회수
- created_at: 작성일
- updated_at: 수정일

카운터 관리
- view_count, like_count, comment_count는 SECURITY DEFINER RPC 함수로 관리
  increment_view_count(post_id) / toggle_like(post_id, user_id) / increment_comment_count(post_id) 등

--------------------------------------------------

3. winterlog_comments 테이블
- 댓글 및 대댓글 정보 저장
- 게시물 상세 페이지의 댓글 영역에서 사용

필드:
- comment_id: 댓글 번호
- post_id: 게시글 번호
- user_id: 작성자 번호
- parent_comment_id: 부모 댓글 번호 (대댓글인 경우)
- content: 댓글 내용
- like_count: 댓글 좋아요 수
- created_at: 작성일

페이지네이션
- 루트 댓글 기준 5개씩 클라이언트사이드 페이지네이션
- 대댓글은 부모 댓글 하단에 항상 표시

--------------------------------------------------

4. winterlog_bookmarks 테이블
- 저장한 게시글 정보 저장
- 마이페이지의 저장한 게시글 목록에서 사용

필드:
- bookmark_id: 북마크 번호
- user_id: 저장한 사용자 번호
- post_id: 저장한 게시글 번호
- created_at: 저장일

--------------------------------------------------

5. winterlog_study_logs 테이블
- 공부 잔디 / 오늘의 목표 기록 저장
- 마이페이지의 공부 잔디와 연속 공부일에서 사용

필드:
- log_id: 공부 기록 번호
- user_id: 사용자 번호
- study_date: 공부 날짜
- goal: 오늘의 목표
- post_count: 작성한 게시글 수
- created_at: 기록일

자동 처리
- 게시글 작성 시 오늘 날짜 레코드 자동 생성 또는 post_count +1 업데이트
- maybeSingle()로 존재 여부 확인 후 insert/update 분기 처리

--------------------------------------------------

6. winterlog_todos 테이블
- 마이페이지 오늘 할 일 저장
- 오늘의 할 일, 진행중, 완료 상태 관리에 사용

필드:
- todo_id: 할 일 번호
- user_id: 사용자 번호
- title: 할 일 제목
- status: todo / progress / done
- created_at: 생성일
- completed_at: 완료일

--------------------------------------------------

7. winterlog_post_reactions 테이블
- 좋아요 / 업보트 / 다운보트 정보 저장
- 한 사용자가 같은 게시글에 중복 반응하지 않도록 관리

필드:
- reaction_id: 반응 번호
- user_id: 반응한 사용자 번호
- post_id: 반응한 게시글 번호
- reaction_type: like / upvote / downvote
- created_at: 반응한 시간

추가 설정
- user_id + post_id + reaction_type 조합을 UNIQUE로 설정
- 한 사용자가 같은 게시글에 동일한 반응을 중복으로 누를 수 없도록 제한

예시
- 같은 사용자가 같은 게시글에 좋아요를 여러 번 누르는 것을 방지
- 같은 사용자가 같은 게시글에 업보트를 중복으로 누르는 것을 방지

--------------------------------------------------

8. winterlog_guestbook 테이블
- 방명록 데이터 저장
- 방명록 페이지 (/guestbook) 에서 사용

필드:
- guestbook_id: 방명록 번호
- user_id: 작성자 번호
- nickname: 작성자 닉네임 (저장 시 스냅샷)
- message: 응원 메시지 (최대 300자)
- emoji: 오늘의 기분 이모지 (🌱 ☁️ 🌙 ✨ 💜 ☕ 🐰)
- support_stars: 응원 온도 (1~5 정수)
- created_at: 작성일

페이지네이션
- 5개씩 서버사이드 페이지네이션
- created_at 내림차순 정렬

--------------------------------------------------

데이터 무결성 관리

- 동일 사용자의 중복 좋아요/업보트 방지를 위해
  user_id + post_id + reaction_type 조합을 UNIQUE로 관리한다.

- 댓글은 parent_comment_id를 사용하여
  대댓글(depth 구조)을 지원한다.

- 게시글 삭제 시 연결된 댓글과 반응 데이터도 함께 정리되도록 설계한다.

- 존재 여부 확인 쿼리는 maybeSingle() 사용 (single()은 데이터 없을 때 406 에러 발생)
